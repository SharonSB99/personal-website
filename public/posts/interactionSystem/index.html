Interaction System

I was the sole engineer in charge of building this new core interaction system in the game.

IGAs will be used in place of Interaction Game Actions.

The old system was bad. - why was the old system bad brief - new system start at the smart object - then go into the gameplay interaction state tree - then go into the front end to the interaction director

Interaction Pain Points of the Past

Interactions used to be executed as game actions where you define the montage to play on the character. This was very inflexible and was unable to handle a change where we had to dynamically assign different kinds of IGAs (Interaction Game Actions) to different objects at the start of a scene. So you may have a character pouring tea at a jukebox, which is not ideal. The cost and reward were also defined on the IGA and had a similar issue to the mismatching animation problem.

Interaction Actors
Figure 1. A MidInteractionActor in a blueprint viewport.

I created a new MidInteractionActor class that houses a Smart Object Component that has a SmartObject Info defined. The Smart Object Info has information on what an interaction looks/does in relation to this actor. Different kinds of Interactions can have different Smart Object definitions depending on placement and behavior. I made a re-usable "single slot in-place" SmartObject definition that follow the pattern of many interactions in the project.

New Smart Object Definitions

A benefit to the refactor is that we could cull alot of repetitive SmartObject definition assets we defined for every single interaction in the project. By making Smart Object Definitions that had common slot placement and character animation behavior, we could simply re-use a generic definition with a state tree that could handle all the specifics of the visualization.

Figure 2. Things like chairs with the same height could use the same smart object definition like the one above.

Of course you can still make a new Smart Object definition for unique interactions that have different requirements and placements. You can make a new one to accomodate a variety of entrances/exits, different number/placement of slots, unique behaviors in specific slots, etc. Until Smart Object definitions become more flexible, we need to define a new definition for interaction actors that are more complicated. An example is a bar with patrons and staff behaving differently and constrained to certain slots (patron side/employee side). The visualization may need different animation assets as well.

Gameplay Interaction State Tree Schemas

a state tree that gets spun up independently from the original context. several kinds of re-usable behavior in our project - pose indefinitely using pose infos - single montage and go - contextual animation with object - contextual animation between two people

Interaction Director Component

add a view of the front end context action menu on click -> sends signal to interaction component that keeps track of valid interactions at any given time
